So what's new this time you ask? Well, nothing. Everything is pretty much the same. In fact, I've used some of the old site's stylesheets to build this site. Unlike all the previous versions, this one is here to stay, for long.

La motivacion #

I've spent some time now (few weeks), searching for a minimal and easy-to-use framework, and in the process of experimentation, I've come across several wonderful tools. There are loads of templating processors and static site generators out there. Some are awesome, while others are just bad. In that unnervingly long list, Jekyll and Gatsby hold the top spot (for obvious reasons). I've used both of them to build the previous versions of this site, and I liked both of them! But (there's always a but), the initial setup is time-consuming. Also, there's a slight chance of your code going rogue as so many dependencies are involved in the process of static site generation (of course the risk varies from site to site). The advantages of using one of them far outweigh the downsides though. Gatsby is unbelievably fast! (Just WOW). Jekyll is simple, easy to use, and also has GitHub Pages support.

I personally don't like having too many dependencies as I believe that my particular use case does not require more than a couple of stylesheets and a script or two. Hence I had to shift my focus towards simpler CSS frameworks like Foundation, Pure, Bootstrap, Bulma, ... (this list is long too). I've used BS & Pure before so I knew the pros and cons of using this kind of framework. Scalability and uniformity are difficult to achieve, but I think it should be just fine for a simple use case like mine. So Powered by twbs it is!

The Unusual stuff #

The navigation and the footer have a weird thing going on. I have two files called nav.html and stickyfooter.html that have navbar and footer elements respectively. As I'm already making use of jQuery, I utilize the load function to load these components into all the pages on this site. This way I can seamlessly edit the nav and footer of all pages by simply editing the component files. 

// Nav Component

$("#navComp").load("/components/nav.html");

// Sticky Footer Component

$("#stickyfooterComp").load("/components/stickyfooter.html");

I'm yet to face the consequences of using this unconventional method (if there are any). There's one more weird thing. I call it the proxy class. Note - I understand that Semantic HTML is a pretty serious concept and is important for accessibility. I've just skipped that part while building this current version of the site (I have plans to work on that part in the later versions). So the people who are serious about the semantic stuff should immediately abandon the site because things are about to get messy.

Maintaining consistency is challenging while using CSS frameworks. There are just too many classes involved and overriding them isn't a good idea all the time. To overcome this issue, I've listed out some standard proxy classes that have a specific role and utilized the addClass function of jQuery to add the actual classes to the class-list of the element. 

$(".pn-html").addClass("h-100");
$(".pn-body").addClass("d-flex flex-column h-100");
$(".pn-main").addClass("flex-shrink-0 pb-3");
$(".pn-page").addClass("pt-4");
$(".pn-pagetitle").addClass("h1");
$(".pn-postdate").addClass("text-muted mb-0");
$(".pn-pagecontent").addClass("pt-4 pagecontent");
$(".pn-subsectitle").addClass("h2 mb-3");
$(".pn-code").addClass("my-4");
$(".pn-footer").addClass("pagefooter mt-auto py-3");
$(".pn-indpostdate").addClass("small text-muted mb-1");
$(".pn-indposttitle").addClass("small text-muted mb-1");
$(".pn-inditem").addClass("pt-4");

As I have to just use the proxy classes, the code looks less bulky, and I can be sure that elements have a proper class-list mapped to them, globally.
